## 1. このガイドについて

このテンプレートでは、AI アシスタントに **決められた日本語フレーズ** を伝えるだけで、決定的な運用モード（workflow）が起動する。モードの意味・分類ルールの正本は `ai/command-router.yml`、各モードの実行手順の正本は `ai/coherence-workflow.yml` である。本ガイドはその使い方を要約する。

```text
あなたの指示 → ai/command-router.yml で分類 → 対応する workflow が動く
```

日常的に使うコマンドは3つだけである。この3つとパターンに一致しない依頼はすべて「運用ルール変更」か「それ以外の雑依頼」として扱われる。

---

## 2. コマンド分類の仕組み

| 分類 | 該当する依頼 | ユーザーへの確認 |
|------|-------------|-----------------|
| **日常コマンド3種** | 下記3種の定型フレーズに一致する依頼 | 原則なし（安全に解いて実行する） |
| **governance_change** | 運用ルール・AI ハーネス・文書統治・仕様駆動運用の変更 | 将来運用に影響するため対話を許可 |
| **shogun_dispatch**（既定） | 上記いずれにも一致しない依頼全般 | 依頼の規模に応じて判断（軽依頼は直接処理） |

分類は AI が `ai/command-router.yml` の `user_patterns` に照らして機械的に行う。あなたがモードを選ぶ必要はない。

## 3. コマンド1: プランの見直し（revise_and_enqueue）

### トリガーフレーズ

```text
◯◯のプランを見直して直近のプランに入れてほしい。詳細設計と要件定義もあわせて見直すこと。
```

`◯◯` に「やりたいこと」を入れる。

### 何が起きるか

`docs/requirements.md`・`docs/design.md` を更新（または「更新不要」の根拠を明記）し、必要なら task spec を作成・更新したうえで、`docs/plan.md` の実行キューへ反映する。plan.md だけの更新では完了と扱わない（仕様駆動開発の鎖を飛ばさない）。

| 更新・生成される文書 | 内容 |
|---------------------|------|
| `docs/requirements.md` | 要件・受入条件の追加/更新 |
| `docs/design.md` | 詳細設計の追加/更新 |
| `docs/plan.md` | Next（実行キュー）への反映 |
| `docs/ai/expectation-ledger.md` | ユーザー意図の記録 |
| `docs/ai/decision-ledger.md` | 判断根拠の記録 |

## 4. コマンド2: バックログの昇格（promote_backlog_to_current_queue）

### トリガーフレーズ

```text
プランのうちバックログの中身をすべて直近のプランに入れて。
```

### 何が起きるか

`docs/plan.md` の Backlog 項目を棚卸しし、重複を除去し、依存関係とリスクで順序付けたうえで、各項目に requirements/design のひもづけ・受入条件・検証コマンド・runtime smoke を補って実行キューへ昇格する。

---

## 5. コマンド3: プランの全実施（execute_current_queue）

### トリガーフレーズ

```text
プランの全実施。
```

### 何が起きるか

`docs/plan.md` の実行キューを先頭から end-to-end で実行する。実装だけで完了とは扱わず、以下を一気通貫で行う。

```text
① plan.md の実行キュー先頭タスクを確認
  │
  ▼
② 実装 → 対象テスト実行 → 静的解析 → 関連 docs/型への伝播 → runtime smoke
  │
  ▼
③ PR 前批判レビュー + 独立監査（completion 条件を満たすまで反復・Backlog 化不可）
  │
  ▼
④ コミット・push・PR 作成（.github/PULL_REQUEST_TEMPLATE.md 準拠）
  │
  ▼
⑤ push 後レビューループ（最大3ラウンド）・CI 最終ゲート
  │
  ▼
⑥ Release Manager が受入条件を逐条判定
  │
  ▼
⑦ 全プラン実行モードなら: マージ → main pull → plan.md / execution-ledger 更新 → 次タスクへ自動継続
   単一タスクのみ実行したい場合は PR 作成時点で完了として報告する
```

修正ループが規定回数を超えた場合や、完了条件を満たせない場合は自動停止し、状況を報告する。

### 全プラン実行モードについて

「プランをすべて実施して」のように継続実行を明示した場合、`.github/full-plan-execution.flag` が作成され、コンテキスト圧縮（compact）をまたいでも全タスク完了まで自動継続する。詳細は `ai/operation-policy.yml` の `compact_survival_contract` と `docs/ai/execution-ledger.md` を参照する。

## 6. 運用ルールの変更（governance_change）

以下のような依頼は、日常コマンドではなく **governance_change** として扱われ、将来運用への影響が大きいため AI からの確認・提案が許可される。

- 運用の見直し（例:「レビューの運用を見直したい」）
- ルールの変更（例:「PR のルールを変更して」）
- AI ハーネスの変更（例:「AI ハーネスの構成を変えたい」）
- 文書構成の整理（例:「文書を整理したい」）
- 仕様駆動開発の運用変更（例:「仕様駆動の運用を変えたい」）

このモードでは `ai/operation-policy.yml`・`ai/context-index.yml`・`ai/document-governance.yml`・`ai/sdd-policy.yml` を確認したうえで、既存ルールの監査 → 変更提案（必要なら質問）→ 入口文書の更新 → 文書棚卸し → harness validator 実行 → 判断記録 → 移行メモ作成、という手順を踏む。

---

## 7. それ以外の依頼（shogun_dispatch）

上記いずれのパターンにも一致しない依頼は、既定で **shogun_dispatch** として処理される。特別なフレーズは不要で、そのまま話しかければよい。

```text
このファイルのバグを修正して
```

```text
CI が失敗しているので直して
```

```text
docs/architecture.md を最新の設計に合わせて更新して
```

依頼の規模に応じて、AI が分解要否を判断する（単純な質問や単発の軽作業は分解せず直接処理する）。コードを変更する場合は、通常の品質チェック（lint・型チェック・テスト・ポリシーチェック）が自動的に実行される。大きな変更や `src/` の変更は PR 経由になることが多い。

運用モデルの詳細は `docs/ai/shogun-operating-model.md` を正本とする。

## 8. コマンドの使い分け早見表

| やりたいこと | 使うコマンド | フレーズ例 |
|-------------|-------------|-----------|
| 新しい要望を要件・設計から見直して計画に入れたい | コマンド1 revise_and_enqueue | 「○○のプランを見直して直近のプランに入れてほしい。詳細設計と要件定義もあわせて見直すこと。」 |
| Backlog の項目をまとめて実行キューに上げたい | コマンド2 promote_backlog_to_current_queue | 「プランのうちバックログの中身をすべて直近のプランに入れて。」 |
| 計画済みタスクを実装からリリースまで進めたい | コマンド3 execute_current_queue | 「プランの全実施。」 |
| 運用ルール・AI ハーネス構成を変えたい | governance_change | 「運用ルールを見直したい」 |
| バグ修正・調査・設定変更など上記以外 | shogun_dispatch | そのまま話しかける |

## 9. よくある質問（FAQ）

### Q1: 3つのコマンドは手動で切り替えるの？

**いいえ**、`ai/command-router.yml` の定型フレーズへの一致で自動的に分類される。フレーズが完全一致しない場合は shogun_dispatch（または実質的に governance_change と判断されれば再分類）に倒れる。

### Q2: 決められたフレーズと少し違う言い方をしたら？

多少の言い回しの違いは AI が意図を汲んで対応することが多いが、確実に動作させるには本ガイドのフレーズをそのまま使うことを推奨する。

### Q3: 実行中に別の指示を出してもいい？

推奨しない。execute_current_queue は多くのステップを順番に実行するため、完了報告を待ってから次の指示を出す方が安全である。

### Q4: 実行キューが空の場合は？

コマンド1またはコマンド2でタスクを追加する。

### Q5: マージは自動でされる？

全プラン実行モードでは、既存のレビューループ・CI・Release Manager 承認をすべて通過した場合に限りマージまで自動で進む。単発実行や明示的なマージ許可（scoped lease）がない場合は、人間の確認・マージが必要である。

### Q6: 運用ルール変更とそれ以外の依頼の境界がわからない場合は？

依頼が「将来の運用にも影響するか」で判断される。判断が難しい依頼は governance_change 側へ倒され、対話で確認される（fail-close）。

## 10. トラブルシューティング

### コマンドが起動しない

| 原因 | 対処 |
|------|------|
| フレーズが定型と一致していない | 本ガイドのフレーズをそのまま使う |
| 実行キューが空 | コマンド1・2でタスクを追加する |
| AI アシスタントの応答上限に達した | しばらく待ってから再度試す |

### 品質チェックが何度も失敗する

1. AI アシスタントにエラー内容を聞く
2. 修正ループが規定回数（既定3ラウンド）を超えた場合は状況報告を待ち、手動で対応する

### 「ポリシー違反」で停止した

1. どのポリシーに違反したか確認する（`docs/policies.md` を参照）
2. 計画自体がポリシーに反していないか見直し、必要ならコマンド1で見直す

### エージェントが応答しない

- `.github/agents/` または使用ハーネスのエージェント定義ファイルが存在するか確認する
- AI アシスタント（拡張機能・CLI）を再起動する

---

> **最後に**: 迷ったら、とりあえず普通に話しかけてください（shogun_dispatch）。計画済みタスクを進めたくなったら「プランの全実施。」——まずはこれだけ覚えれば十分です。
